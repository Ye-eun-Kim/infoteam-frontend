# Git과 Github

## Git이란?

프로그램을 개발할 때는 일반적으로 여러 기능을 여러 사람이 함께 작업합니다. 따라서 각자 작업한 것을 합치고 작업물을 관리해야 하는데 이것을 용이하게 해주는 것이 바로 버전 관리 시스템(Version Control System, VCS)입니다. 예를 들어 VCS가 변경된 코드를 기억해주기 때문에 개발자는 수정된 부분을 간편하게 확인할 수 있으며 수정 버전을 이전 버전으로 롤백 할 수도 있습니다.
우리는 VCS 중에서 가장 넓게 사용되고 있는 Git을 사용할 것입니다. Git은 메인 저장소를 복사한 개인 저장소에서 버전 관리를 하고, 그걸 메인 저장소에 올리는 방식으로 작업이 이루어집니다.

## Github란?

Git은 로컬(여러분의 pc)에서 버전 관리 시스템을 운영하는 방식입니다. Github는 클라우드 서버를 저장소로 제공하여 Git으로 작업한 것을 클라우드 방식으로 관리하도록 돕는 버전 관리 시스템입니다. Github를 통해 Git으로 관리하는 자료를 다른 사람과 공유하거나 백업할 수 있고, 다른 사람이 Github에 올린 자료를 복제할 수도 있습니다. 협업을 위한 기능이 잘 구축되어 있어 공동의 프로젝트를 진행할 때 유용합니다.

### Git, Github를 이용하여

* 공동 프로젝트에서 코드 리뷰 가능(누가, 언제, 무엇을 수정했는지)
* Github을 이용해 자신의 local에서 개발한 내용 공유 가능
* Github의 커뮤니티 기능 활용. (Issue를 통한 버그 제시 등)
* visual studio, IntelliJ, Android Studio 등 다양한 IDE에서 git 연동 제공

### Git 시작하기
Git에서 소스 코드가 저장되고 버전 관리가 적용되는 공간을 저장소라고 합니다. 여러분의 컴퓨터에 저장된 저장소를 로컬 저장소라 하고, 인터넷이나 네트워크에 올라가 있는 저장소를 원격 저장소라 합니다. 일반적으로 공동 작업물을 보관하는 원격 저장소를 orgin이라 합니다. 원격 저장소 origin이 우리가 작업하는 소스 코드의 근원이자 결과물을 올리는 곳이죠. 이 원격 저장소 관리 서비스를 제공하는 서비스가 Github입니다.

#### 주요 Git 명령어


#### `git clone`

이 명령어를 실행하면 원격 저장소를 다운 받아 로컬 저장소에 옮겨올 수 있습니다. 이 경우 자동으로 원격저장소가 orgin이라는 이름으로 설정되어 있습니다.

```bash
git clone git@github.com:dodok8/silver-bassoon.git
```

#### `git init`

저장소를 만들고 싶은 디렉토리로 이동하여 이 명령어를 실행하면 숨김 설정이 되어 있는 .git 디렉토리가 생깁니다. .git디렉토리에는 저장소에 필요한 뼈대 파일(Skeleton)이 들어있습니다. 저장소에 파일을 추가하여 개발을 진행합니다.

#### `git remote`

컴퓨터에 저장되어 있는 로컬 저장소와 네트워크(혹은 다른 위치에 있는) 저장소를 연결해 줍니다. 이 저장소가 원격 저장소가 됩니다. 아래 명령어는 주소에 나와있는 저장소를 origin이라는 이름의 원격 저장소로 지정하겠다는 의미입니다.

```bash
git remote add origin git@github.com:dodok8/silver-bassoon.git
```

#### `git status`

git은 파일을 크게 두 종류로 나누어 관리합니다. 하나는 변경 사항을 계속 추적하는 Tracked, 변경 사항 추적에서 제외하는 Untracked입니다. 이 명령어를 실행하면 Tracked 파일들의 변경사항을 알려줍니다. 여러 옵션을 추가하여 더 자세한 정보를 볼 수 있습니다. 일단 기본적으로 저장소 내 모든 파일은 Tracked입니다.

파일을 Untracked로 바꾸기 위해선 `.gitignore` 파일을 만듭니다. `.gitignore` 파일 안에 디렉토리나 파일의 상대 경로를 적어주면 자동으로 Untracked 상태가 됩니다. 특정 확장자를 제외하고 싶을 경우, 예를 들어 xs파일을 제외하고 싶으면 `*.xs`라 적으면 됩니다.

이 gitignore는 직접 만들수도 있지만 요즘 사용되는 대부분의 개발환경 툴에는 `.gitignore`를 자동으로 만들어 주는 옵션이 들어있고, [gitignore.io](https://www.gitignore.io/) 같은 곳에서 여러분의 환경에 맞는 프리셋을 찾을 수도 있습니다.


#### `git add`

이 명령어는 저장하기 원하는 파일을 stage 하는 명령어입니다. stage(staging)는 버전 관리 시스템에 기록할 파일을 골라 staging area에 저장해두는 일입니다.
용도에 맞게 인자를 설정해주면 됩니다. 예를 들어 작업 디렉토리의 변경 내용의 일부만 staging area에 넘기고 싶을 때는 수정한 파일이나 디렉토리의 경로를 인자로 넘깁니다.
```bash
git add <파일/디렉토리 경로>
```
이 외에도 `git add -A`, `git add .`, `git add -p` 등이 있습니다.

*Git의 공간은 아래와 같이 크게 4가지로 구분됩니다. (참고: https://seonkyukim.github.io/git-tutorial/git-add-commit/#working-directory-staging-area-git-directory)
* Working directory: 작업 중인 로컬 컴퓨터의 공간. 이곳에서 파일 수정.
* Staging area/Index: 워킹 디렉터리에서 작업한 내역을 Git directory로 저장하기 전에 작업 내역의 복사본을 담아두는 영역.
* Git directory(.git): staging area의 파일들의 영구적인 스냅샷을 저장하는 장소.
* Stash: 위 세 공간과 별개로 임시 저장이 가능한 영역.


#### `git commit`

이 명령어는 stage 해서 고른 파일들(staging area에 저장됨)을 commit이라는 단위로 묶어서 git directory에 기록해줍니다. 이때 기록되는 건 stage 시점의 파일이므로 stage 이후의 수정 사항을 commit 하고자 할 때는 다시 stage를 해주어야 합니다.
commit을 할 때는 이 commit에서 무엇을 했는지 메세지를 남깁니다. 기본적으로 git에서 만들어주기는 하지만 `git commit -m '메시지 내용'` 으로 커밋 메세지를 남겨주시길 바랍니다.

#### `git push`

commit들을 원격 저장소에 올립니다. 만약 로컬 저장소와 원격저장소 사이에 충돌이 있을 경우(다른 사람의 커밋이 먼저 올라가 있는 경우) 다음 명령어인 `git pull`을 먼저 실행해야 합니다. 강제로 push 하는 방법도 있지만 사용하지 않는 것이 강력하게 권고됩니다.


#### `git pull`

원격 저장소의 데이터를 받아와 로컬 저장소의 데이터와 병합합니다. 하지만 만약 로컬 저장소와 원격 저장소에서 변경 대상이 같다면 충돌이 발생하며 수동으로 해결해야 합니다.

git pull = git fetch(원격 저장소에서 데이터 가져옴) + git merge(병합)

#### `git fetch`

이 명령을 실행하면 원격 저장소의 최신 commit을 로컬로 가져올 수 있습니다. 하지만 `git pull`과 달리 작업 중인 로컬 저장소와 병합하진 않습니다. 병합을 원할 때는 `git merge`, `git rebase` 등의 명령어를 사용합니다.


### 잘못된 커밋 고치기


#### `git reset <옵션> <커밋번호>`

이 명령을 실행하면 작업이 커밋 번호 시점으로 돌아가고 그 이후 커밋들을 삭제합니다.
옵션은 `--mixed`, `--soft`, `--hard` 등이 있으며 필요에 따라 사용합니다.

만약 리셋을 사용하기 전에 푸시한 내역이 있을 경우 당연히 충돌이 발생합니다. `git push --force`로 강제로 덮어 씌울 수도 있지만 혼자 작업하는 브랜치가 아닌 경우에는 reset을 쓰지 말고 다음에 말할 revert를 사용하길 바랍니다.


#### `git revert <커밋번호>` 

이 명령을 실행하면 커밋 번호 시점으로 작업물을 되돌릴 수 있습니다. reset과는 다르게 revert에 대한 커밋도 일종의 커밋으로 간주하여 기록하고 작업을 되돌립니다.


### 브랜치 이해하기

git에는 브랜치라는 개념이 있습니다. 여러 명이 동시에 한 저장소에서 작업을 하면 계속 충돌을 해결하면서 진행해야 합니다. 이를 해결하기 위해 각자 브랜치라는 것을 만들어서 거기서만 작업을 하고 나중에 합치는 것이죠. 분업을 효율적으로 가능케 해줍니다.


#### `git branch`

브랜치 목록을 보고 만들게 해주는 명령어입니다.


#### `git checkout`

현재 이용 중인 브랜치를 바꾸는(이를 checkout이라고 합니다.) 명령어입니다. `git checkout -b <브랜치명>`을 하면 새로운 브랜치를 만드는 동시에 체크아웃을 진행합니다.


#### `git stash`

커밋이 안 된 수정된 파일이 있을 경우에는 브랜치를 바꿀 수 없습니다. 그 때 사용하는 것이 stash입니다. 일종의 임시 저장 개념으로, 현재까지 수정된 내용을 묶어서 기록해줍니다. stash를 하고 나서는 체크아웃을 진행할 수 있습니다. 다시 원래 브랜치로 돌아와서 `git stash apply`로 다시 꺼내 쓸 수 있습니다.


### 작업내용 합치기

이 내용은 앞서 `git pull`, `git fetch`에서 다룬 내용과 유사하지만 이번에는 브랜치 간의 병합을 보겠습니다. 커맨드라인에 이 명령어를 쳐서 바로 합칠 수도 있지만, 아래 Github 단락에서 다루는 Pull & Request를 **꼭** 진행해 주셨으면 합니다.

작업내용이 합해진 다음, 필요없는 브랜치는 삭제하면 됩니다.


#### `git merge`

여러 브랜치에서 작업한 내용을 합치는 커밋을 만들어 줍니다. 리베이스에 비해 최종 그래프가 복잡해지나 각 브랜치에서 작업한 내용을 더 직관적으로 볼 수 있고 충돌이 더 적게 일어납니다. 다른 브랜치끼리 합칠 때는 머지를 추천합니다.


#### `git rebase`

여러 브랜치에서 진행된 커밋을 한 브랜치에서 작업한 것처럼 순서를 조정해줍니다. 훨씬 깔끔하게 보이는 장점이 있으나, 각 브랜치에서 무엇을 했는지 파악하기 힘들어지는 단점이 있고, 충돌이 더 많이 일어납니다.

### Github 활용하기

#### Pull & Request

다른 브랜치나 포크한 저장소에서 작업한 내용을 원래 저장소에 병합하고 싶지만 권한이 없을 때 만드는 신청입니다. github의 다른 기능과 연동해서 테스트 코드 작성 및 통과 여부, 충돌 여부 등을 손쉽게 파악할 수 있고 팀 프로젝트의 작업물 병합 과정에서 최종결정권자가 정리할 수 있습니다.


#### Issue tracker

버그나 의문점 등이 생겼을 때 이를 작성하고 보고하는 게시판입니다. 인포팀 프론트 엔드는 업무를 진행할 때 이슈를 작성한 뒤 작업에 들어갑니다. 풀 리퀘스트와 마찬가지로 다른 기능과 연동해서 단순한 게시판 이상의 능력을 가질 수 있습니다. 현재 슬랙과 연동이 되어 있기 때문에 github에 이슈가 올라오면 자동으로 슬랙에 반영됩니다.


#### Github Pages

Github 저장소에서 바로 정적인 페이지를 만들 수 있게 해줍니다. 기본 옵션으로 jekyll을 사용해서 진행하도록 되어 있습니다.


#### Github Action

일정 간격, 혹은 git과 관련된 특정행동(push, pull, pull request, issue 등)을 감지했을 때 등록된 세팅에 따라 코드를 실행해 주는 기능입니다. 아래의 Secret과 연동해서 보안에 민감한 내용은 환경변수로 넘겨 줄 수도 있습니다. 한 가지 단점은 이 코드에서 직접적으로 DB에 접근하거나 새로운 파일을 만드는 작업같은 건 힘듭니다. 그렇지만 이메일 발송이나 테스트 자동화 정도는 훌룡하게 해줄 수 있습니다.


#### Github Secret

보안상 민감한 내용을 암호화 해서 보관해주는 서비스입니다. 예를 들어 DB 관리자 계정같은 정보를 암호화해서 넘겨줄 수 있습니다.


## Git을 편하게 써보자

사실 Git을 커맨드로만 사용하는 것은 많이 불편합니다. 여러가지 클라이언트가 많이 나와있죠. 추천하는 건 [GitKraken](https://www.gitkraken.com/)입니다. Source Tree같은 다른 훌룡한 앱도 많지만 인포팀 멤버들이 이것을 많이 들 사용 중입니다.
